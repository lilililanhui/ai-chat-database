# SDD Feature Skill

## 概述

SDD (Spec Driven Development) Feature Skill 是一个用于指导 AI 按照规范驱动开发模式新增功能的技能模块。

## 什么是 SDD？

Spec 驱动开发是一种先编写规范文档、再进行代码实现的开发方法。核心理念：

1. **规范先行** - 在编码之前，先定义清楚功能的需求、接口和行为
2. **文档即代码** - 规范文档是项目的重要组成部分，需要维护和版本控制
3. **AI 辅助实现** - AI 根据规范文档生成符合要求的代码

## 使用方法

### 方式一：自动触发

当你向 AI 描述新增功能的需求时，AI 会自动加载本 Skill：

```
用户: 我想添加一个消息搜索功能
AI: [加载 SDD Feature Skill]
    好的，让我按照 SDD 流程来帮你实现这个功能...
```

### 方式二：手动激活

你可以在对话中明确要求使用 SDD 模式：

```
用户: 请使用 SDD 模式帮我新增一个导出功能
```

本项目的规范文件存放在 `specs/` 目录：

## 最佳实践

### 1. 需求足够具体

❌ "添加一个导出功能"  
✅ "添加一个将对话导出为 Markdown 文件的功能，包含消息时间戳"

### 2. 先读后写

让 AI 先读取现有的 Spec 文件，了解项目上下文：

```
用户: 先读取一下项目的架构规范，然后帮我添加 XXX 功能
```

### 3. 分步验证

对于复杂功能，可以分步验证：

1. 先确认 Spec 文件是否正确
2. 再确认接口实现是否符合规范
3. 最后确认 UI 是否符合预期

### 4. 保持文档同步

- 每次功能变更都更新对应的 Spec 文件
- 重大变更更新 CHANGELOG
- 定期检查 Spec 与代码的一致性

## 常见问题

### Q: Spec 文件需要多详细？

A: 取决于功能复杂度。简单功能可以只写概述和接口定义；复杂功能建议写完整的用户故事、组件结构和测试用例。

### Q: 是否每个小改动都需要 Spec？

A: 不需要。Bug 修复、样式调整等小改动不需要专门的 Spec。只有新功能或重大重构才需要。

### Q: Spec 和代码不一致怎么办？

A: 以实际需求为准。如果代码实现有合理的偏差，应该更新 Spec 以保持一致。
