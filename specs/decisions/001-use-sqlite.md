# ADR-001: 使用 SQLite 作为本地数据库

## 状态

已接受 (Accepted)

## 背景

AI 知识库助手需要一个本地数据存储方案来保存：
- 会话信息和配置
- 聊天消息历史
- 知识库文档索引

需求约束：
- 必须支持本地运行，无需外部数据库服务
- 需要支持 Electron 桌面应用打包
- 数据需要持久化，应用重启后可恢复
- 查询性能要求适中，数据量预计在 10MB 级别

## 决策

选择 **SQLite**（使用 `better-sqlite3` 库）作为本地数据库解决方案。

## 理由

### 选择 SQLite 的优势

1. **零配置**
   - 无需安装额外的数据库服务
   - 单文件存储，易于管理

2. **单文件存储**
   - 整个数据库是一个 `.db` 文件
   - 方便备份和迁移（直接复制文件即可）

3. **性能优秀**
   - 对于本地应用场景，性能完全满足需求
   - 支持索引优化查询

4. **Electron 兼容**
   - `better-sqlite3` 是原生模块，性能优于纯 JS 实现
   - 支持同步 API，代码更简洁

5. **成熟稳定**
   - SQLite 是世界上使用最广泛的数据库引擎
   - 经过数十年的生产验证

### better-sqlite3 vs sqlite3

| 特性 | better-sqlite3 | sqlite3 |
|------|----------------|---------|
| API 风格 | 同步 | 异步 (回调) |
| 性能 | 更快 | 较慢 |
| 代码复杂度 | 简单 | 复杂 |
| 维护状态 | 活跃 | 活跃 |

选择 `better-sqlite3` 因为：
- 同步 API 更适合简单的本地应用
- 性能更好
- 代码更简洁易维护

## 后果

### 积极影响

✅ 简化部署流程 - 无需配置数据库服务  
✅ 减少依赖 - 无需 Docker 或数据库安装  
✅ 易于备份 - 复制文件即可  
✅ 开发简单 - 同步 API 降低复杂度  

### 消极影响

⚠️ 不支持多进程并发写入 - 单写入者模式  
⚠️ 不适合大规模数据 - 建议数据量 < 1GB  
⚠️ 原生模块编译 - Electron 打包需要重新编译  

### 风险缓解

1. **并发写入问题**
   - 当前应用是单用户本地应用，不存在并发写入场景
   - 如果未来需要支持，可以使用 SQLite 的 WAL 模式

2. **数据量增长**
   - 实现定期清理旧数据的功能
   - 监控数据库文件大小

3. **Electron 编译**
   - 使用 `electron-rebuild` 工具
   - 在 CI/CD 中配置正确的构建环境

## 替代方案

### IndexedDB

- 优点：浏览器原生支持，无需原生模块
- 缺点：仅浏览器环境可用，API 复杂

### LevelDB

- 优点：嵌入式，性能好
- 缺点：不支持 SQL，查询能力有限

### PostgreSQL

- 优点：功能强大，支持复杂查询
- 缺点：需要部署数据库服务，对本地应用过于复杂

### JSON 文件

- 优点：最简单，无依赖
- 缺点：无法高效查询，数据量大时性能差
